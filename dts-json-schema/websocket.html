Websocket¶

Endpoint¶

We use single-use websocket url, which can be obtained using the REST API.

Captioner Ready¶

When a captioner is ready to handle the conversation, Ava sends a ready message to the client, which contains:

  • type (string): ready.

Exampple:

{
  "type": "ready"
}

Json Schema:

{
  "$schema": "http://json-schema.org/draft-06/schema#",
  "$id": "websocket-message:ready",
  "title": "Ready message",
  "description": "Message sent when captioner is ready",
  "type": "object",
  "properties": {
    "type": {
      "const": "ready"
    }
  },
  "required": [
    "type"
  ]
}

At this point, client can start to send audio messages.

note: in dev mode (if the conversation has been started with "dev": true), the ready message will be sent without having a real captioner ready.

Audio¶

In order to stream the audio, the client sends audio with the audio to transcribe. No audio message should be sent before receiving the ready event.

  • type (string): audio.
  • audio (Array<integer>): one audio chunk (must match the encoding, the length and the sampleRateHertz indicated in connection setup).
  • timestamp (integer): timestamp of the beginning of audio.

Exampple:

{
  "type": "audio",
  "audio": [
    22,
    0,
    -17,
    112
  ],
  "timestamp": 1531587765
}

Json Schema:

{
  "$schema": "http://json-schema.org/draft-06/schema#",
  "$id": "websocket-message:audio",
  "title": "Audio message !",
  "type": "object",
  "properties": {
    "type": {
      "const": "audio"
    },
    "audio": {
      "type": "array",
      "description": "one audio chunk (must match the encoding, the length and the sampleRateHertz indicated in connection setup).",
      "items": {
        "type": "integer"
      }
    },
    "timestamp": {
      "type": "integer",
      "description": "timestamp of the beginning of audio."
    }
  },
  "required": [
    "type",
    "audio",
    "timestamp"
  ]
}

Transcript¶

We return transcripts to the client in transcript messages:

  • type (string): transcript.
  • blocId (integer): id of the trancript.
  • transcript (string): transcript text
  • isFinal (boolean): if true, then no more transcript associated to this blocId will be published.

Exampple:

{
  "type": "transcript",
  "blocId": 1531587700,
  "transcript": "Hello, World.",
  "isFinal": true
}

Json Schema:

{
  "$schema": "http://json-schema.org/draft-06/schema#",
  "$id": "websocket-message:transcript",
  "title": "Transcript message",
  "type": "object",
  "properties": {
    "type": {
      "const": "transcript"
    },
    "blocId": {
      "type": "integer",
      "description": "id of the trancript."
    },
    "transcript": {
      "type": "string",
      "description": "transcript text"
    },
    "isFinal": {
      "type": "boolean",
      "description": "if :code:`true`, then no more transcript associated to this :code:`blocId` will be published."
    }
  },
  "required": [
    "type",
    "transcript",
    "blocId",
    "isFinal"
  ]
}

If two transcripts are sent with the same blocId, only the last one should be considered. For each blocId, we will send a transcript with a isFinal field set to true.

Please note that the blocId will be ordered: if a blocId is smaller than an other one, then the associated transcripts are ordered in the same way.

Ping / pong¶

We expect to receive at least one message every 10s. If during the connection (captioner ready or not), you think you may have a large period without sending message, you must send a ping. We will send back a pong.

End Connection¶

When the conversation is finished, the client should send a end message over the websocket:

  • type (string): end.

Exampple:

{
  "type": "end"
}

Json Schema:

{
  "$schema": "http://json-schema.org/draft-06/schema#",
  "$id": "websocket-message:end",
  "title": "End message",
  "type": "object",
  "properties": {
    "type": {
      "const": "end"
    }
  },
  "required": [
    "type"
  ]
}

Ava will consider as error all the ws connections which are closed without end message.

Errors¶

For various reasons, we may send error messages. Client should expect that the underlying WebSocket connection will be closed by us after sending an error message. Error message contains:

  • type (string): error.
  • error (object): An error object (see Errors).

Exampple:

{
  "type": "error",
  "error": {
    "name": "VALIDATION_ERROR",
    "fields": {
      "audio": "Audio does not match the required length"
    }
  }
}

Json Schema:

{
  "$schema": "http://json-schema.org/draft-06/schema#",
  "$id": "websocket-message:error",
  "title": "Error message",
  "description": "Error message sent by ava backend to the client",
  "type": "object",
  "properties": {
    "type": {
      "const": "error"
    },
    "error": {
      "$ref": "error",
      "description": "An error object (see :doc:`error`)."
    }
  },
  "required": [
    "type",
    "error"
  ]
}
